# C# Server Learning Project

## Goal

이 프로젝트는 C#으로 학습용 서버를 직접 만들어 보면서 서버 구조, I/O 멀티플렉싱, 테스트 주도 개발(TDD), 코어 라이브러리와 컨텐츠 계층 분리를 익히는 것을 목표로 한다.

가장 중요한 목표는 **TDD 개발 흐름을 실제로 경험하는 것**이다. 기능을 먼저 크게 구현하는 방식이 아니라, 실패하는 테스트를 통해 요구사항을 작게 정의하고, 최소 구현으로 통과시키고, 이후 리팩토링하는 흐름을 반복한다.

## Session Rule

이 세션에서는 사용자가 명시적으로 코드 추가, 설정 변경, 파일 수정을 요청하지 않는 한 파일에 쓰지 않는다.

프롬프트가 모호한 경우에는 작업을 가정해서 바로 진행하지 않고, 먼저 한 번 질문한다. 답변을 받은 뒤에만 다음 단계로 진행한다.

## Plan

1. 학습 목표를 작은 기능 단위로 나눈다.
2. 각 기능마다 먼저 테스트할 수 있는 요구사항을 문장으로 정의한다.
3. 실패하는 테스트를 작성한다.
4. 테스트를 통과시키는 최소한의 구현만 작성한다.
5. 테스트가 통과한 뒤 중복, 책임 분리, 이름, 구조를 리팩토링한다.
6. 같은 흐름을 반복하며 서버의 기능을 점진적으로 확장한다.

## Core Topics

### 1. I/O Multiplexing

서버는 여러 클라이언트의 연결과 요청을 동시에 처리해야 한다. 이 프로젝트에서는 C#의 소켓 API와 비동기 I/O 모델을 이용해 다중 연결을 처리하는 구조를 학습한다.

처음부터 고성능 서버를 목표로 하기보다는 다음 순서로 접근한다.

1. 단일 클라이언트 연결 처리
2. 여러 클라이언트 연결 처리
3. 비동기 accept/read/write 흐름 분리
4. 세션 객체 도입
5. 패킷 수신 버퍼와 메시지 경계 처리
6. 서버 종료와 리소스 정리

멀티스레드 또는 비동기 콜백이 포함되는 순간부터는 공유 상태 접근, 세션 수명, 연결 해제 중복 처리에 대한 스레드 안전성 검토가 필요하다.

### 2. TDD

TDD는 테스트를 나중에 붙이는 방식이 아니라, 구현 전에 요구사항을 검증 가능한 형태로 바꾸는 개발 방식이다.

기본 흐름은 다음과 같다.

1. Red: 아직 구현되지 않은 동작에 대한 실패 테스트를 작성한다.
2. Green: 테스트를 통과시키는 최소 구현을 작성한다.
3. Refactor: 테스트가 통과하는 상태를 유지하면서 구조를 개선한다.

이 프로젝트에서는 특히 다음 장점을 체감하는 것을 목표로 한다.

- 요구사항을 작고 명확하게 쪼개는 습관을 만든다.
- 구현 전에 사용 방식과 인터페이스를 먼저 고민하게 한다.
- 리팩토링 시 기존 동작이 깨졌는지 빠르게 확인할 수 있다.
- 네트워크 코드처럼 재현이 어려운 영역도 테스트 가능한 단위로 분리하게 만든다.

### 3. Core Library and Content Layer

서버의 공통 기능은 코어 라이브러리로 만들고, 실제 게임/채팅/컨텐츠 로직은 이를 상속 또는 조합해서 구현하는 구조를 목표로 한다.

예상 구조는 다음과 같다.

```text
Core
- SocketServer
- Session
- PacketBuffer
- PacketDispatcher
- Serializer

Content
- ChatSession
- ChatPacketHandler
- Room
- User

Tests
- Core.Tests
- Content.Tests
```

초기에는 상속보다 조합이 더 단순할 수 있다. 따라서 코어 라이브러리는 가능한 한 특정 컨텐츠를 모르게 만들고, 컨텐츠 계층이 세션 이벤트나 패킷 핸들러를 연결하는 방향을 우선 검토한다.

### 4. Redis

Redis는 필수 목표가 아니라 부가 목표로 둔다. 서버 기본 구조와 TDD 흐름이 안정된 뒤, 다음 상황이 생기면 도입을 검토한다.

- 로그인 토큰 또는 세션 상태 저장
- 랭킹, 카운터, TTL이 필요한 임시 데이터
- 서버 간 pub/sub 메시지 전달
- 캐시 계층 실습

Redis 도입 시에는 실제 Redis 서버에 의존하는 통합 테스트와, Redis 접근 로직을 추상화한 단위 테스트를 분리한다.

## Suggested First Milestones

1. 테스트 프로젝트를 만들고 가장 작은 순수 로직 테스트부터 시작한다.
2. `PacketBuffer`를 TDD로 구현한다.
3. `Session`의 수신 데이터 처리 흐름을 테스트 가능한 형태로 분리한다.
4. 로컬 TCP echo 서버를 만든다.
5. 여러 클라이언트 접속 테스트를 추가한다.
6. 컨텐츠 계층에서 간단한 채팅 메시지를 처리한다.
7. 필요해진 시점에 Redis 연동을 검토한다.

## TDD Practice Rule

새 기능을 추가할 때는 항상 다음 질문에 먼저 답한다.

1. 이 기능의 관찰 가능한 동작은 무엇인가?
2. 네트워크 없이 테스트할 수 있는 부분은 어디인가?
3. 실패 테스트를 가장 작게 만들려면 어떤 입력과 출력이 필요한가?
4. 테스트를 통과시키는 최소 구현은 무엇인가?
5. 리팩토링 후에도 같은 테스트가 의미를 유지하는가?

## Teaching Approach

이 프로젝트에서 에이전트는 단순히 코드를 대신 작성하는 역할이 아니라, 사용자가 C# 서버 구조와 TDD 흐름을 체감하도록 돕는 선생 역할을 한다.

진행 방식은 다음 원칙을 따른다.

- 먼저 요구사항을 테스트 가능한 문장으로 바꾼다.
- 구현보다 테스트 설계를 먼저 논의한다.
- 너무 큰 단계를 작은 단위로 쪼갠다.
- 테스트가 실패하는 이유를 설명한 뒤 최소 구현을 제안한다.
- 구현 후에는 테스트가 설계를 어떻게 보호하는지 되짚는다.
- 비동기, 소켓, 멀티스레드 코드에서는 스레드 안전성을 별도로 확인한다.
